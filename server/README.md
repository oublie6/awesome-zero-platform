# Server

The repository's first runnable server is a go-zero REST process under `apps/app-api`. The current foundation includes standard response envelopes, application errors, request IDs, panic recovery, structured access logging, configurable CORS, baseline security headers, request body limits, reusable MySQL and Redis lifecycle packages, dependency-aware readiness, and current schema/seed definitions for local development.

## Prerequisites

- Go `1.25.8`
- `goctl` `1.10.1`

Install `goctl`:

```bash
go install github.com/zeromicro/go-zero/tools/goctl@v1.10.1
```

## Structure

- `apps/app-api/app.api` — go-zero API contract source
- `apps/app-api/etc/main-api.yaml` — safe local development configuration
- `apps/app-api/internal/types` — generated request and response types
- `apps/app-api/internal/handler/routes.go` — generated route registration
- `apps/app-api/internal/logic` — generated logic scaffolds with handwritten health responses
- `apps/app-api/internal/bootstrap` — handwritten configuration loading, startup, and HTTP foundation integration
- `foundation/apperrors` — reusable application error model
- `foundation/cache` — focused Redis configuration, startup checks, and lifecycle ownership
- `foundation/database` — focused MySQL configuration, startup checks, lifecycle ownership, and transaction execution
- `foundation/requestid` — request ID validation, generation, and context propagation
- `foundation/readiness` — bounded dependency readiness aggregation
- `foundation/response` — response envelopes and go-zero response integration
- `foundation/httpmiddleware` — focused middleware for request IDs, access logging, recovery, CORS, security headers, and body limits
- `database/schema/current.sql` — current complete database schema
- `database/seed/development.sql` — optional development seed content

Generated files remain marked by goctl headers. Files marked `Code generated by goctl. DO NOT EDIT.` are regenerated from `app.api`. Files marked `Code scaffolded by goctl. Safe to edit.` are the intended handwritten extension points.

## Commands

From the repository root:

```bash
make generate
make fmt
make test
make deps-up
make schema-apply
make seed-apply
make integration-test
make run
make deps-down
```

The generation command run by `make generate` is:

```bash
cd server && goctl api go --api apps/app-api/app.api --dir apps/app-api --style gozero
```

If you prefer to work directly inside `server`:

```bash
go test ./...
find . -name '*.go' -type f -print0 | xargs -0 gofmt -w
go run ./apps/app-api -f apps/app-api/etc/main-api.yaml
```

The root dependency commands are:

```bash
make deps-up
make deps-down
make deps-reset
make schema-apply
make seed-apply
make integration-test
```

- `make deps-up` starts pinned local MySQL and Redis containers and waits for both health checks to pass.
- `make deps-down` stops the local dependency containers without keeping extra runtime artifacts.
- `make deps-reset` deliberately recreates the local dependency containers and drops their transient data.
- `make schema-apply` applies `server/database/schema/current.sql` to the local MySQL instance.
- `make seed-apply` applies the optional development seed at `server/database/seed/development.sql`.
- `make integration-test` runs real dependency tests serially with `APP_API_INTEGRATION=1` and the `integration` build tag.

## Identity foundation

- Identity is currently an internal platform capability only and lives under `server/platform/identity`.
- The module owns `identity_accounts` and `identity_password_credentials`; other modules must not query those tables directly.
- Account IDs use application-generated UUIDv7 values instead of MySQL auto-increment sequencing.
- Supported account states are `active` and `disabled`.
- No public registration, login, token, session, password-change, or account-management HTTP endpoint is exposed in this phase.

Identity normalization is deterministic and intentionally conservative:

- `username` is trimmed, must be 3-32 ASCII characters using letters, digits, `.`, `_`, or `-`, and is matched case-insensitively by a stored lowercase key.
- `email` is trimmed, limited to ASCII, validated as a plain address, and matched case-insensitively by a stored lowercase key.
- `phone` is trimmed and must already be in explicit E.164 form such as `+14155550123`; no locale-specific rewriting is applied.
- At least one of `username`, `email`, or `phone` is required when creating or updating an account profile.

Password credentials are stored separately from account profile data:

- Plaintext passwords are never stored, logged, seeded, or returned.
- Hashing uses Argon2id with explicit defaults: memory `65536` KiB, iterations `3`, parallelism `2`, salt length `16`, key length `32`.
- The committed test suite uses bounded test parameters without replacing the production defaults.
- Password inputs are validated before hashing and must be between `8` and `128` bytes.

The current schema remains rebuildable from `server/database/schema/current.sql`. No development seed account, default administrator, role, permission, or password is committed in this phase.

## Startup

`make run` starts the API on the host and port configured in `apps/app-api/etc/main-api.yaml`.

The committed development configuration is:

```yaml
Name: main-api
Host: 0.0.0.0
Port: 8888
HTTP:
  MaxBodyBytes: 1048576
  RequestID:
    HeaderName: X-Request-Id
    MaxLength: 64
  SecurityHeaders:
    ContentTypeOptions: nosniff
    FrameOptions: DENY
    ReferrerPolicy: no-referrer
MySQL:
  Addr: 127.0.0.1:3306
  Database: awesome_zero_platform
  User: app_local
  Password: local-dev-only-mysql-password
  Charset: utf8mb4
  ParseTime: true
  Location: UTC
  TimeZone: +00:00
Redis:
  Addr: 127.0.0.1:6379
  Password: local-dev-only-redis-password
Readiness:
  Timeout: 2s
Startup:
  ConnectivityTimeout: 3s
```

Startup fails with a non-zero exit if the config file is missing, if required HTTP or dependency settings are invalid, or if MySQL or Redis cannot be reached within the configured bounded startup timeout.

The committed MySQL and Redis credentials are development-only values for the pinned local containers. Do not reuse them outside local development, and do not commit real credentials or runtime data.

go-zero REST handles graceful shutdown by default. Sending `SIGINT` or `SIGTERM` stops the process cleanly.

Resource ownership is explicit:

- `app-api` opens MySQL first, then Redis.
- If Redis initialization fails after MySQL starts, MySQL is closed before startup returns the error.
- Normal shutdown stops the HTTP server and then closes Redis followed by MySQL. Close operations are safe to call more than once.

## HTTP conventions

Ordinary successful API responses use a standard JSON envelope:

```json
{
  "code": "OK",
  "message": "success",
  "requestId": "caller-id-or-generated-id",
  "data": {
    "status": "ok"
  }
}
```

Application and internal errors use the same shape with `data: null` and semantically correct HTTP status codes:

```json
{
  "code": "PARAM_INVALID",
  "message": "invalid parameter",
  "requestId": "01ABCDEF...",
  "data": null
}
```

Reusable foundation error codes:

- `PARAM_INVALID`
- `UNAUTHORIZED`
- `FORBIDDEN`
- `NOT_FOUND`
- `CONFLICT`
- `REQUEST_TOO_LARGE`
- `INTERNAL_ERROR`

## Request IDs

- The server reads request IDs from the `X-Request-Id` request header.
- Caller-supplied IDs are preserved only when they are non-empty, at most 64 characters, and contain only letters, digits, `.`, `_`, or `-`.
- Missing or invalid values are replaced with a generated compact ID.
- The effective request ID is returned in the `X-Request-Id` response header and in standard response envelopes.

## Body limit, recovery, and access logging

- `HTTP.MaxBodyBytes` configures the maximum request body size. The committed default is `1048576` bytes.
- Oversized requests return HTTP `413` with code `REQUEST_TOO_LARGE`.
- Unexpected panics are recovered at the HTTP boundary, logged server-side with stack traces, and returned as HTTP `500` with the safe `INTERNAL_ERROR` envelope.
- Each completed request emits a structured access log with `requestId`, `method`, `path`, `statusCode`, `elapsed`, and `clientAddress`.
- `clientAddress` currently records the direct peer address from `RemoteAddr`. Forwarded headers are not trusted in this phase.

## CORS and security headers

- CORS is disabled by default and enabled only when `HTTP.CORS.Enabled` is set.
- Supported CORS settings are `AllowedOrigins`, `AllowedMethods`, `AllowedHeaders`, `ExposedHeaders`, and `AllowCredentials`.
- Wildcard origins must not be combined with credentialed requests; startup validation rejects that configuration.
- Matched routes return these baseline security headers:
  - `X-Content-Type-Options: nosniff`
  - `X-Frame-Options: DENY`
  - `Referrer-Policy: no-referrer`

## Health endpoints

`/health/live` and `/health/ready` deliberately keep their minimal probe-friendly payloads instead of using the standard response envelope. They still receive request ID propagation, recovery, access logging, CORS handling, security headers, and other globally safe middleware.

Liveness:

```bash
curl -s http://127.0.0.1:8888/health/live
```

Expected response:

```json
{"status":"ok"}
```

Readiness:

```bash
curl -s http://127.0.0.1:8888/health/ready
```

Expected response:

```json
{"status":"ready"}
```

Readiness performs bounded MySQL and Redis health checks and returns HTTP `200` with `{"status":"ready"}` only when both dependencies are available. If either dependency is unavailable, readiness returns HTTP `503` with `{"status":"unready"}`.

Readiness responses intentionally remain minimal and do not expose DSNs, passwords, internal addresses, raw driver errors, or stack traces.

## Schema and seed policy

- `server/database/schema/current.sql` stores the current complete schema for this phase of the project.
- Incremental migration history is intentionally not committed yet.
- Temporary database upgrade SQL must not be committed.
- `server/database/seed/development.sql` is optional and contains only non-business development seed content.

Apply the deterministic local workflow from the repository root:

```bash
make deps-reset
make schema-apply
make seed-apply
make integration-test
make run
```

## Troubleshooting

- If `make run` fails during startup, check that `make deps-up` completed successfully and rerun `make schema-apply`.
- If readiness returns `503`, verify that both local dependency containers are healthy with `docker compose -f deploy/local/docker-compose.yml ps`.
- If you want a clean local state, use `make deps-reset` and then reapply schema and seed.
- Keep integration and verification commands serial on low-memory machines; the committed Make targets already avoid parallel Go test execution.
