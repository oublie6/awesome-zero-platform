# Server

The repository's first runnable server is a go-zero REST process under `apps/app-api`. The current foundation includes standard response envelopes, application errors, request IDs, panic recovery, structured access logging, configurable CORS, baseline security headers, and request body limits.

## Prerequisites

- Go `1.25.8`
- `goctl` `1.10.1`

Install `goctl`:

```bash
go install github.com/zeromicro/go-zero/tools/goctl@v1.10.1
```

## Structure

- `apps/app-api/app.api` — go-zero API contract source
- `apps/app-api/etc/main-api.yaml` — safe local development configuration
- `apps/app-api/internal/types` — generated request and response types
- `apps/app-api/internal/handler/routes.go` — generated route registration
- `apps/app-api/internal/logic` — generated logic scaffolds with handwritten health responses
- `apps/app-api/internal/bootstrap` — handwritten configuration loading, startup, and HTTP foundation integration
- `foundation/apperrors` — reusable application error model
- `foundation/requestid` — request ID validation, generation, and context propagation
- `foundation/response` — response envelopes and go-zero response integration
- `foundation/httpmiddleware` — focused middleware for request IDs, access logging, recovery, CORS, security headers, and body limits

Generated files remain marked by goctl headers. Files marked `Code generated by goctl. DO NOT EDIT.` are regenerated from `app.api`. Files marked `Code scaffolded by goctl. Safe to edit.` are the intended handwritten extension points.

## Commands

From the repository root:

```bash
make generate
make fmt
make test
make run
```

The generation command run by `make generate` is:

```bash
cd server && goctl api go --api apps/app-api/app.api --dir apps/app-api --style gozero
```

If you prefer to work directly inside `server`:

```bash
go test ./...
find . -name '*.go' -type f -print0 | xargs -0 gofmt -w
go run ./apps/app-api -f apps/app-api/etc/main-api.yaml
```

## Startup

`make run` starts the API on the host and port configured in `apps/app-api/etc/main-api.yaml`.

The committed development configuration is:

```yaml
Name: main-api
Host: 0.0.0.0
Port: 8888
HTTP:
  MaxBodyBytes: 1048576
  RequestID:
    HeaderName: X-Request-Id
    MaxLength: 64
  SecurityHeaders:
    ContentTypeOptions: nosniff
    FrameOptions: DENY
    ReferrerPolicy: no-referrer
```

Startup fails with a non-zero exit if the config file is missing or if required settings such as `Name`, `Host`, `Port`, request ID settings, body-size limits, or contradictory CORS options are invalid.

go-zero REST handles graceful shutdown by default. Sending `SIGINT` or `SIGTERM` stops the process cleanly.

## HTTP conventions

Ordinary successful API responses use a standard JSON envelope:

```json
{
  "code": "OK",
  "message": "success",
  "requestId": "caller-id-or-generated-id",
  "data": {
    "status": "ok"
  }
}
```

Application and internal errors use the same shape with `data: null` and semantically correct HTTP status codes:

```json
{
  "code": "PARAM_INVALID",
  "message": "invalid parameter",
  "requestId": "01ABCDEF...",
  "data": null
}
```

Reusable foundation error codes:

- `PARAM_INVALID`
- `UNAUTHORIZED`
- `FORBIDDEN`
- `NOT_FOUND`
- `CONFLICT`
- `REQUEST_TOO_LARGE`
- `INTERNAL_ERROR`

## Request IDs

- The server reads request IDs from the `X-Request-Id` request header.
- Caller-supplied IDs are preserved only when they are non-empty, at most 64 characters, and contain only letters, digits, `.`, `_`, or `-`.
- Missing or invalid values are replaced with a generated compact ID.
- The effective request ID is returned in the `X-Request-Id` response header and in standard response envelopes.

## Body limit, recovery, and access logging

- `HTTP.MaxBodyBytes` configures the maximum request body size. The committed default is `1048576` bytes.
- Oversized requests return HTTP `413` with code `REQUEST_TOO_LARGE`.
- Unexpected panics are recovered at the HTTP boundary, logged server-side with stack traces, and returned as HTTP `500` with the safe `INTERNAL_ERROR` envelope.
- Each completed request emits a structured access log with `requestId`, `method`, `path`, `statusCode`, `elapsed`, and `clientAddress`.
- `clientAddress` currently records the direct peer address from `RemoteAddr`. Forwarded headers are not trusted in this phase.

## CORS and security headers

- CORS is disabled by default and enabled only when `HTTP.CORS.Enabled` is set.
- Supported CORS settings are `AllowedOrigins`, `AllowedMethods`, `AllowedHeaders`, `ExposedHeaders`, and `AllowCredentials`.
- Wildcard origins must not be combined with credentialed requests; startup validation rejects that configuration.
- Matched routes return these baseline security headers:
  - `X-Content-Type-Options: nosniff`
  - `X-Frame-Options: DENY`
  - `Referrer-Policy: no-referrer`

## Health endpoints

`/health/live` and `/health/ready` deliberately keep their minimal probe-friendly payloads instead of using the standard response envelope. They still receive request ID propagation, recovery, access logging, CORS handling, security headers, and other globally safe middleware.

Liveness:

```bash
curl -s http://127.0.0.1:8888/health/live
```

Expected response:

```json
{"status":"ok"}
```

Readiness:

```bash
curl -s http://127.0.0.1:8888/health/ready
```

Expected response:

```json
{"status":"ready"}
```
